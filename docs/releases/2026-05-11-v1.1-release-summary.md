# AnyKnews V1.1 Release Summary

## Release Goal

V1.1 turns AnyKnews from a fixed set of homepage cards into a cost-aware, user-configurable source catalog.

The main product decision was to keep the app lightweight: no login, no database, browser-local subscriptions, server memory cache plus compact disk cache, and source fetching limited to subscribed or explicitly requested sources.

## Requirement Evolution

1. Expanded the source catalog from the original default board into a 28-source catalog.
2. Kept the default subscribed sources unchanged so the homepage still opens with the familiar board.
3. Replaced sources that were unsuitable for normal reading:
   - Replaced 机器之心 with AIbase because 机器之心 articles required payment.
   - Changed 雪球 to 热门话题.
   - Changed 汽车之家 to 今日焦点 from `https://www.autohome.com.cn/all/`.
   - Changed 36氪 to `资讯-推荐`.
4. Changed the cost model away from full-page eager refresh:
   - Page open refreshes subscribed first-page data.
   - Low-cost sources refresh when stale.
   - Medium/high-cost sources only refresh when the last successful refresh is older than 10 minutes.
   - Card pagination loads more items from the single-source API on demand.
   - Manual card refresh targets one source only.
5. Added compact server cache and failed-fetch backoff:
   - Disk cache stores source metadata, title, summary, link, metrics, timestamps and diagnostics.
   - It does not store full HTML, article bodies, images or raw connector payloads.
6. Improved the reading board:
   - Each source card shows 8 items per page and supports pagination up to the fetched item limit.
   - External links open in a new tab.
   - Category navigation filters the current board view.
   - Single-card refresh returns focus to that card instead of resetting the whole page.
7. Redesigned subscription management into a top-level page:
   - `订阅设置` no longer shows news cards.
   - The left side is the subscribed-source list.
   - The right side is the full source catalog grouped by category.
   - Source cards show icon, name, intro, recommendation score and preview action.
   - Category sections use stronger visual grouping.
   - Sources inside each category sort by recommendation score from high to low.
   - Left and right panels scroll independently on desktop.
8. Simplified V1.1 personalization:
   - Source subscription and drag ordering remain active.
   - Favorite, keyword, ignore and hidden-source controls are temporarily removed from the visible UI.
   - Older browser-local fields are preserved for future versions.

## Connector Scope

The V1.1 catalog target is 28 sources:

- AI 资讯: 量子位, AIbase, Anthropic News
- 技术: GitHub Trending, V2EX, Hacker News, InfoQ, 少数派
- 广义资讯: 知乎热榜, 今日头条热榜, 澎湃新闻, 微博热搜, 抖音热点, 小红书热点, 豆瓣
- 科技资讯: 36氪, 虎嗅, 晚点, 钛媒体
- 娱乐: B 站热门, 游民星空
- 金融: 雪球, 财新, 界面, 华尔街见闻, 东方财富
- 汽车: 汽车之家, 懂车帝

Connectors prefer official APIs, RSS, stable public JSON endpoints, DailyHotApi, 60s API and RSSHub before custom HTML parsing. Sources with upstream instability keep typed fallback data so the UI remains usable.

## Verification

V1.1 was locally verified on the development preview at `http://localhost:3014/`.

Required release checks:

```bash
npx tsx scripts/verify-source-catalog.ts
npx tsx scripts/verify-refresh-policy.ts
npx tsx scripts/verify-source-cache-store.ts
npx tsc --noEmit
npm run lint
npm run build
```

Manual checks:

- Default subscribed board loads.
- Category navigation filters source cards.
- Source card pagination works.
- Single-source refresh does not reset the whole page focus.
- `订阅设置` is top-level and contains only subscription operations.
- Left subscribed list and right catalog panel scroll independently.
- Source categories show visually distinct catalog groups.
- Category source order follows recommendation score.

## Release Governance Correction

During the first cloud publish, the Tencent Cloud working tree was temporarily switched directly to the `v1.1` version branch. That proved the deployment but violated the production branch rule.

The corrected release rule is:

- `v1.1` remains the development and acceptance branch.
- The accepted `v1.1` branch is merged into `main`.
- Tencent Cloud production is redeployed from `main`.
- Future production releases must use `main` or an immutable release tag only.

This correction is documented in:

- `docs/codex-workspace-cicd-standard.md`
- `AGENTS.md`
- `docs/working-agreements.md`
- `docs/tencent-cloud-deploy.md`
- `.github/workflows/ci.yml`

The release path is also guarded by executable checks:

- `scripts/guard-production-release.sh` refuses production release unless the worktree is clean, on `main`, and exactly matches `origin/main`.
- `scripts/deploy-production.sh` pulls `main`, runs the release guard, validates Compose, rebuilds only the app service, and verifies the health endpoint.

## Known Follow-Ups

- Notification endpoints remain implemented but should not be scheduled online until digest templates are refined.
- Some public upstreams may still fall back under anti-crawl or RSSHub/60s service instability.
- Browser-local preferences are acceptable for V1.1; server-side sync can be revisited if account/login becomes necessary.
